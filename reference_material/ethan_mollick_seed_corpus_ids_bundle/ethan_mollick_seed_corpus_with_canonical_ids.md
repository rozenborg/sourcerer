# Ethan Mollick-Style Seed Corpus with Canonical IDs

**Purpose:** identifier-enriched seed corpus for Semantic Scholar research monitoring.

**Important caveat:** the original seed corpus intentionally included exact papers, title variants, search phrases, category seeds, and Mollick-adjacent expansion items. Not every row has, or should have, a canonical paper identifier. Rows that are not single identifiable scholarly records are explicitly flagged rather than forced into false matches.

## Identifier strategy

- Prefer `semantic_scholar_paper_id` and `semantic_scholar_corpus_id` when verified.
- Add DOI, arXiv, SSRN, PubMed, PMCID, NBER, OpenReview, World Bank handle, and RePEc IDs when available.
- Use `s2_api_lookup_key` for programmatic retrieval. Semantic Scholar Graph API accepts lookup keys such as `CorpusID:<id>`, `DOI:<doi>`, `ARXIV:<id>`, `PMID:<id>`, and direct paper IDs. SSRN rows are usually best resolved through their DOI form `10.2139/ssrn.<id>`.

## Status codes

| Status | Meaning |
|---|---|
| `S2_RESOLVED` | Semantic Scholar paper ID and/or corpus ID verified. |
| `EXTERNAL_ID_ONLY` | A robust external ID was found, but no Semantic Scholar ID was verified in this pass. |
| `DUPLICATE_OR_VARIANT` | Search variant or duplicate row; identifiers inherited from canonical row. |
| `AMBIGUOUS_OR_CATEGORY` | Not a single resolvable paper title; usually a topic/category/search phrase. |
| `UNRESOLVED_LOOKUP_REQUIRED` | Likely a real title, but canonical ID still needs API/title verification. |

## Recommended ingestion policy

1. Use `S2_RESOLVED` rows as direct positive Semantic Scholar seeds.
2. Resolve `EXTERNAL_ID_ONLY` rows with the Semantic Scholar Graph API using `s2_api_lookup_key`; many DOI/arXiv records will map to S2 IDs.
3. Keep `DUPLICATE_OR_VARIANT` rows for query expansion, but do not double-weight them as independent papers.
4. Do not ingest `AMBIGUOUS_OR_CATEGORY` rows as papers. Use them as topic/query scaffolding only.
5. Send `UNRESOLVED_LOOKUP_REQUIRED` rows through the included bulk resolver and then require title/author/year validation before seeding.
